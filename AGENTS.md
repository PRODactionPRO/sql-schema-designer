# AGENTS POLICY — Branching, PR, Deploy

Этот файл обязателен для всех ассистентов и разработчиков, работающих в репозитории.

## 1) Базовые правила веток

- Долгоживущие ветки: только `main` и `develop`.
- `main` = production (только стабильный код).
- `develop` = интеграционная ветка (Dev Holding).
- Любая задача делается в отдельной feature-ветке от `develop`:
  - `feature/<scope>-<short-name>`
  - пример: `feature/sidebar-dnd`, `feature/api-auth`

## 2) Правильный поток PR

- Запрещено делать PR `feature/* -> main`.
- Обязательный путь:
  1. `feature/* -> develop`
  2. review + green checks
  3. merge в `develop`
  4. отдельный PR `develop -> main`
  5. deploy после green checks на `main`

## 3) Работа в параллельных чатах

- Один чат = одна задача = одна ветка.
- Запрещено создавать локальные копии репозитория, соседние папки проекта или `git worktree` без прямой команды пользователя.
- Если ассистент видит накопленные изменения и не понимает, как безопасно продолжать, он обязан остановиться, описать ситуацию пользователю и ждать ответа. Самостоятельно создавать копию проекта, переносить изменения в другую папку или обходить проблему через новый worktree запрещено.
- Если пользователь явно разрешил параллельную работу через `git worktree`, путь, ветка и причина создания worktree должны быть проговорены до выполнения команды.
- Не смешивать в одном чате задачи разных подсистем (Canvas/API/Sidebar).
- Не переиспользовать старую feature-ветку для новой задачи.

## 4) Правила перед началом работы

Перед любыми изменениями ассистент обязан:

1. Показать текущую ветку (`git branch --show-current`).
2. Подтянуть актуальный remote (`git fetch origin`).
3. Убедиться, что задача не делается в `main` и не в `develop`.
4. Создать новую feature-ветку от `origin/develop` (если ветка не создана заранее).

## 5) Правила перед PR

Перед созданием PR ассистент обязан:

1. Проверить target branch (для feature всегда `develop`).
2. Прогнать локальные проверки:
   - `npm run typecheck`
   - `npm run lint`
   - `npm run test` (если есть изменения в логике)
3. Убедиться, что в PR нет чужих/случайных файлов.
4. Добавить краткое описание: что сделано, как проверено, риски.

## 6) Деплой-политика

- Деплой только из `main`.
- Деплой запускается только после успешных CI-checks.
- Если после merge в `main` CI красный — сначала фикс, потом деплой.
- Нельзя деплоить из feature-веток и из `develop`.

## 7) Что запрещено

- Прямые коммиты в `main` и `develop` для feature-задач.
- PR в `main` в обход `develop`.
- Force-push в `main`/`develop`.
- Выполнять деплой при красных checks.
- Тихо менять целевую ветку PR без явного подтверждения пользователя.

## 8) Быстрый чеклист ассистента

- Я точно не в `main` и не в `develop`?
- Ветка создана от `origin/develop`?
- PR идет в `develop`?
- Checks зеленые?
- Только после этого можно готовить `develop -> main` и deploy.
